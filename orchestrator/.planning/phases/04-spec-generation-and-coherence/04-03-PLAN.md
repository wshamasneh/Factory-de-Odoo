    1. Create commands/odoo-gsd/plan-module.md following the exact pattern of discuss-module.md:
       - Frontmatter: name "odoo-gsd:plan-module", description about generating spec.json, argument-hint "{module_name}"
       - allowed-tools: Read, Bash, Write, Task, AskUserQuestion
       - Context section explaining what the command does
       - Objective: "Generate a complete spec.json for the specified module from CONTEXT.md and registry context"
       - After this command note: "Review spec.json, then run `/odoo-gsd:generate-module {module}` to generate code"
       - execution_context pointing to @~/.claude/odoo-gsd/workflows/plan-module.md

    2. Create odoo-gsd/workflows/plan-module.md with the locked 10-step flow:

       ## Step 1: Validate Module Status
       - Read module_status.json via: `node "$HOME/.claude/odoo-gsd/bin/odoo-gsd-tools.cjs" module-status get "${MODULE}" --raw --cwd "$(pwd)"`
       - Verify status is "planned" (or allow re-run if "spec_approved" with user confirmation to overwrite)
       - If module not found, error with helpful message

       ## Step 2: Check Existing Spec
       - Check if `.planning/modules/${MODULE}/spec.json` exists
       - If exists, use AskUserQuestion: "spec.json already exists. Overwrite, skip, or abort?"

       ## Step 3: Load Module Context
       - Read `.planning/modules/${MODULE}/CONTEXT.md`
       - If missing, error: "Run /odoo-gsd:discuss-module ${MODULE} first"

       ## Step 4: Load Decomposition Entry
       - Read `.planning/decomposition.json` and find the entry for this module
       - Extract: module name, description, depends list, tier

       ## Step 5: Spawn Researcher Agent
       - Spawn Task(subagent_type="odoo-gsd-module-researcher") with module context
       - Agent produces `.planning/modules/${MODULE}/RESEARCH.md`
       - Read the produced RESEARCH.md

       ## Step 6: Build Tiered Registry
       - If model_registry.json exists and is non-empty:
         Run `node "$HOME/.claude/odoo-gsd/bin/odoo-gsd-tools.cjs" registry tiered-injection "${MODULE}" --raw --cwd "$(pwd)"`
       - If registry is empty or missing (first module in project): use empty tiered result `{"direct": {}, "transitive": {}, "rest": []}`

       ## Step 7: Spawn Spec Generator Agent
       - Spawn Task(subagent_type="odoo-gsd-spec-generator") with:
         - Module name
         - CONTEXT.md content
         - RESEARCH.md content
         - Decomposition entry (name, description, depends, tier)
         - Config odoo block (read from .planning/config.json)
         - Tiered registry JSON (for _available_models section)
       - Agent writes spec.json to `.planning/modules/${MODULE}/spec.json`
       - Read and validate the produced spec.json (valid JSON, has all 12 top-level keys)

       ## Step 8: Run Coherence Check
       - Run: `node "$HOME/.claude/odoo-gsd/bin/odoo-gsd-tools.cjs" coherence check --spec ".planning/modules/${MODULE}/spec.json" --registry ".planning/model_registry.json" --raw --cwd "$(pwd)"`
       - Parse the JSON coherence report
       - Save report to `.planning/modules/${MODULE}/coherence-report.json`

       ## Step 9: Spawn Reviewer Agent (Presentation)
       - Spawn Task(subagent_type="odoo-gsd-spec-reviewer") with:
         - Coherence report JSON
         - Path to spec.json
         - Module name
       - Agent presents human-readable summary

       ## Step 10: Human Approval
       - If coherence status is "fail":
         Use AskUserQuestion: "Coherence check found violations. Options: approve_anyway, revise (re-run from Step 5), abort"
       - If coherence status is "pass":
         Use AskUserQuestion: "Spec looks good. Options: approve, revise, abort"
       - On "approve" or "approve_anyway":
         Run: `node "$HOME/.claude/odoo-gsd/bin/odoo-gsd-tools.cjs" module-status transition "${MODULE}" spec_approved --raw --cwd "$(pwd)"`
         Print confirmation message
       - On "revise": Go back to Step 5 (researcher)
       - On "abort": Stop workflow, no status change

    IMPORTANT: Each step should have clear error handling. If any step fails, present a clear error and stop.
    IMPORTANT: Use the exact CLI invocation patterns shown in the interfaces section.
    IMPORTANT: For Task() spawning, use the subagent_type pattern from discuss-module.md workflow.
  </action>
